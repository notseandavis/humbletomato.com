BMAD Method

Structured AI-assisted development with specific agent roles for different phases of software creation.

What Is BMAD?

BMAD is a complementary approach to spec-driven development that organizes AI agent work through specialized roles rather than a single autonomous loop.

While the Matrix Methodology focuses on specifications as source of truth and the Ralph Loop provides autonomous execution, BMAD explores how different agent roles can collaborate on complex development tasks.

Core Concept

The central idea: different phases of development benefit from different agent configurations — different context windows, different prompts, different specializations.

Rather than a single agent handling everything from architecture to implementation to testing, BMAD distributes work across specialized roles:

  • Builder — Generates implementation code from specifications
  • Maintainer — Handles refactoring, optimization, and technical debt
  • Analyzer — Reviews code quality, architecture alignment, test coverage
  • Designer — Focuses on user experience, interface patterns, accessibility

Each role has different constraints, different context priorities, and different success criteria.

How It Complements Matrix

The Matrix Methodology provides the what and how through structured specs and architecture patterns. BMAD provides the who — which agent role should handle which type of work.

Think of Matrix as the governance layer (specs, templates, quality gates) and BMAD as the execution layer (role-based agent orchestration).

Early Insights

While detailed documentation is pending, here are some early learnings:

Role Specialization Reduces Context Bloat

A single agent trying to handle architecture, implementation, and testing simultaneously struggles with context management. Specialized roles can maintain focused context windows with only the information relevant to their phase.

Handoffs Need Structure

The transition between roles (Builder → Analyzer, for example) requires structured artifacts. Just like the Matrix uses specs and fix_plan.md, BMAD needs role-specific handoff documents.

Not Every Task Needs Every Role

Simple feature additions might only need Builder. Complex refactors might need Maintainer + Analyzer. New user-facing features might need Designer + Builder + Analyzer. The orchestration layer needs to be smart about which roles to activate.
