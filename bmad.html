BMAD Method

Build More Architect Dreams. An agent-as-code framework that installs specialized AI personas, workflows, and templates into your project for structured development.

Source: GitHub · Docs · Open source, no paywalls

What It Actually Is

BMAD installs markdown and YAML files into your project that define AI personas, workflows, and templates. Not a runtime, not a SaaS, not a plugin. It's context engineering: structured files that shape how LLMs behave during development.

Installation creates two folders: _bmad/ (framework: agents, workflows, config) and _bmad-output/ (generated artifacts: PRD, architecture, stories).

npx bmad-method install

Works with Claude Code, Cursor, and Codex CLI.

The Core Innovation: Document-as-Context Handoffs

Each development phase produces markdown documents that become structured context for the next phase's agents. The Analyst creates a product brief that the PM reads when writing the PRD. The Architect reads the PRD when designing the system. The Developer reads the architecture doc when implementing stories.

Analyst  -> product-brief.md
PM       -> PRD.md (reads product-brief)
Architect -> architecture.md (reads PRD)
PM       -> epics/*.md (reads PRD + architecture)
SM       -> story files (reads epics)
Developer -> working code (reads story + architecture)

Each handoff is a fresh chat session. All continuity lives in the files, not conversation history. This prevents context window pollution but means agents can't learn from mistakes within a session.

Agent Personas

Each agent is a markdown file with YAML frontmatter defining its role, persona, commands, and dependencies. Agents are scoped to specific context, not given everything:

AgentRoleContext Loaded
Analyst ("Mary")Requirements, research, brainstormingExisting briefs
PM ("John")PRD, epics, product strategyProduct brief, existing PRD
ArchitectSystem design, ADRs, tech decisionsPRD, project-context.md
Developer ("James")Implementation, code reviewStory file, architecture, project-context
Scrum MasterSprint planning, story creationPRD, architecture, epics

BMad OrchestratorCoordinates workflows, can transform into other agentsAll

Three Planning Tracks

BMAD offers three levels of ceremony. You pick the track; the system doesn't auto-detect.

TrackForArtifacts
Quick FlowBug fixes, simple featuresTech-spec only
BMad MethodProducts, platformsPRD + Architecture + UX
EnterpriseCompliance, multi-tenantFull spec suite + security + DevOps

The bmad-help skill recommends a track based on your description, but the choice is manual. "Scale-adaptive intelligence" is the marketing term for having three predefined tracks available.

Party Mode

Multiple agent personas collaborate in one session. The BMad Orchestrator decides which persona responds to each message. Useful for brainstorming and surfacing tradeoffs from different perspectives.

Under the hood, this is a single LLM role-playing multiple characters, not independent agents with separate reasoning. It works because LLMs are good at maintaining character consistency. Don't confuse it with actual multi-agent systems.

Strengths

StrengthDetail
Structured handoffsDocument-as-context model ensures agents always have relevant, scoped input.
Comprehensive lifecycleDiscovery through deployment. ~20 core workflows across 4 phases.
ModularAdd test architecture, game dev, or creative modules as needed.
Team-friendlyRole separation maps to real team structures. Audit trails via artifacts.
Analyst agentRequirements elicitation techniques described as "one of the best personal findings in recent years" by a real user.

Tradeoffs

LimitationDetail
Ceremony overheadFull BMAD flow for a simple feature adds hours of planning that may not improve the outcome.
Hallucination compoundingEach agent builds on the previous agent's output. Errors in the PRD propagate to architecture, stories, and code.
Greenfield biasAssumes clean-slate architecture creation. Brownfield projects need manual workarounds.
No parallelismAgent handoffs are inherently sequential. No wave-based parallel execution like GSD.
Learning curve12+ agents, ~20 workflows, multiple modules. Quick Flow exists but feels like an afterthought.
PO validation gapProduct Owner validation produces a report but no agent references it. Creates a workflow dead-end.

When to Use BMAD

Complex greenfield project where you don't yet know exactly what to build. You need the planning discipline to figure it out. You have (or simulate) a team and want structured role separation. You need audit trails for enterprise environments.

When to Look Elsewhere

Solo developer who already knows what they want. Bug fixes or small features (use Quick Flow at minimum, or GSD). Brownfield projects needing surgical changes. You want speed over comprehensive documentation.
