Code Diffusion Model

Your specs must be as detailed as the system is complex. There is no shortcut.

The Analogy

A diffusion model generates images by resolving noise into coherent output. Give it "a photo" and you get something blurry and generic. Give it "a barista pouring latte art in a sunlit cafe, morning light through steam, shallow depth of field" and you get something specific and detailed.

The output can't contain more information than the input. A vague prompt produces a vague image. A detailed prompt produces a detailed image. The complexity of the result is bounded by the information you provide.

Prompt: "a website"
Output: Generic template. Looks like a website. Doesn't look like YOUR website.

Prompt: "e-commerce platform, OAuth2 auth, Redis-cached cart,
         Stripe payments, saga-based order processing,
         role-based access for customers vs merchants"
Output: Specific implementation. Looks like what you described.

Conservation Law

Complexity is conserved. You can move it around — from code to specs, from specs to architecture docs, from architecture docs to tribal knowledge. But you can't destroy it.

Simple spec    → Agent guesses → Complex debugging
Detailed spec  → Agent resolves → Working system
                                  
The work is the same. The question is when you do it:
before generation (in the spec) or after (in debugging).

When your spec says "handle authentication," you haven't saved any work. You've moved it from "writing the spec" to "debugging why the agent used session cookies when you needed JWT." The complexity showed up anyway. It just showed up at the worst possible time.
