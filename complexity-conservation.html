Conservation of Complexity

Your specs must be as detailed as the system is complex. There is no shortcut.

The Diffusion Model

A diffusion model generates images by resolving noise into coherent output. Give it "a photo" and you get something blurry and generic. Give it "a barista pouring latte art in a sunlit cafe, morning light through steam, shallow depth of field" and you get something specific and detailed.

The output can't contain more information than the input. A vague prompt produces a vague image. A detailed prompt produces a detailed image. The complexity of the result is bounded by the information you provide.

Prompt: "a website"
Output: Generic template. Looks like a website. Doesn't look like YOUR website.

Prompt: "e-commerce platform, OAuth2 auth, Redis-cached cart,
         Stripe payments, saga-based order processing,
         role-based access for customers vs merchants"
Output: Specific implementation. Looks like what you described.

You Have to Build the Whole System

Here's the trap. You're building the cart service. You think: "I'll focus the context on cart logic. Keep it simple."

But the cart service needs to know:

  • Auth — whose cart is this? How are users scoped? JWT subject claim? Session cookie?
  • Inventory — can you add out-of-stock items? When do you check availability?
  • Payments — how does checkout hand off? What's the cart-to-order transition?
  • State — Redis for speed? Postgres for durability? Both? Cache invalidation strategy?
  • Pricing — discounts applied at cart level or checkout? Coupon validation happens where?

Every "other" system touches the cart. You can't restrict the context to just cart logic because cart logic doesn't exist in isolation. The boundaries between components are load-bearing walls, not decorative trim.

What you think you're building:
┌──────────┐
│   Cart   │
└──────────┘

What you're actually building:
┌──────────┬──────────┬──────────┬──────────┐
│   Auth   │   Cart   │ Inventory│ Payments │
├──────────┼──────────┼──────────┼──────────┤
│ JWT      │ Add/     │ Stock    │ Checkout │
│ scoping  │ Remove   │ check    │ handoff  │
├──────────┼──────────┼──────────┼──────────┤
│ Session  │ Pricing  │ Reserve  │ Cart →   │
│ binding  │ rules    │ logic    │ Order    │
└──────────┴──────────┴──────────┴──────────┘

A diffusion model can't render just the nose without knowing where the eyes are. Every pixel resolves relative to every other pixel. The system is the unit of generation, not the component.

Conservation Law

Complexity is conserved. You can move it around — from code to specs, from specs to architecture docs, from architecture docs to tribal knowledge. But you can't destroy it.

Simple spec    → Agent guesses → Complex debugging
Detailed spec  → Agent resolves → Working system
                                  
The work is the same. The question is when you do it:
before generation (in the spec) or after (in debugging).

When your spec says "handle authentication," you haven't saved any work. You've moved it from "writing the spec" to "debugging why the agent used session cookies when you needed JWT." The complexity showed up anyway. It just showed up at the worst possible time.

The Engineer's Wrong Instinct

Engineers are trained to isolate. Encapsulate. Reduce scope. "I'm building the cart service, so I'll scope the context to cart stuff. Less noise. More focused."

Good instinct for writing code. Wrong instinct for composing context.

The agent building the cart needs to see the auth model, the payment flow, the inventory rules. Not because it's building those things, but because cart decisions are shaped by those things. How the cart scopes to a user depends on how auth works. How checkout hands off depends on how payments work. How "add to cart" validates depends on how inventory works.

# Scoped context (engineer's instinct)
"Build the cart service. Here's the cart spec."
→ Cart works perfectly in isolation.
→ Breaks the moment it touches auth, payments, inventory.

# Full system context
"Build the cart service. Here's the cart spec,
 the auth model, the payment flow, the inventory rules."
→ Cart integrates correctly on first generation.

The Context Matrix is built around this idea. Every row needs every column. You can't give the agent one row in isolation and expect it to resolve correctly, because each cell is constrained by the cells around it.

You don't get to build a component. You build a component within a system, or you debug the system the agent imagined around it.
