Continuous Evolution

Spec-driven development for live systems. Work from diffs, not regeneration.

# Greenfield: build from complete spec
Spec → Agent → System

# Evolution: build from the delta
Spec(v1) → Spec(v2) → Diff → Agent → Targeted Changes

The Matrix Methodology regenerates from specs. That breaks when you have customers, production data, and API contracts. The fix: agents work from spec version diffs, not whole specs.

Expand-Migrate-Contract

# Phase 1: Expand — add new alongside old
ALTER TABLE users ADD COLUMN email_v2 VARCHAR(255);
# Code writes to BOTH, reads from old

# Phase 2: Migrate — backfill and switch
UPDATE users SET email_v2 = normalize(email);
# Code reads from new, writes to both

# Phase 3: Contract — remove old (one-way door)
ALTER TABLE users DROP COLUMN email;
# Rollback no longer possible

Slow by design. Greenfield sprints, evolution walks.

Feature Flags as Rollback

if feature_flags.is_enabled("new_email"):
    send_via_new(msg)
else:
    send_via_legacy(msg)

Not true rollback — both paths deployed. But you can disable without redeploying.

Agent Rules

  • Never DROP without explicit instruction
  • Never assume data loss is acceptable
  • Flag migrations that lock production tables
  • Preservation is default: anything not in diff stays unchanged

A project shifts from greenfield to evolution at the first real user or irreplaceable data. Before: regenerate freely. After: diffs only.

Hyrum's Law: every observable behavior becomes depended-on. Gall's Law: complex systems that work evolved from simple systems that worked. Both say: evolve from diffs, not regeneration.
