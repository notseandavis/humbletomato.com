Continuous Evolution

Extending spec-driven development to live products with real users, real data, and real constraints. Work from diffs, not regeneration.

Greenfield: Agent builds from complete spec. Spec → Agent → System.

Evolution: Agent builds from the delta. Spec(v1) → Spec(v2) → Diff → Agent → Targeted Changes.

The Matrix Methodology regenerates from specs. That stops working when you have paying customers, production data, and API contracts you can't break.

The fix: same spec-driven approach, but the agent works from the diff between spec versions, not the whole spec.

Expand-Migrate-Contract

The safe pattern for structural database changes:

# Phase 1: Expand — add new alongside old
ALTER TABLE users ADD COLUMN email_v2 VARCHAR(255);
# Code writes to BOTH columns, reads from old

# Phase 2: Migrate — backfill and switch
UPDATE users SET email_v2 = normalize(email);
# Code reads from new, still writes to both

# Phase 3: Contract — remove old (one-way door)
ALTER TABLE users DROP COLUMN email;
# Rollback no longer possible

This is slow by design. In greenfield you sprint. In evolution you walk.

Versioned Workflow State

User starts a workflow Monday. You deploy Tuesday. They resume Wednesday. Their state assumes the old world.

{
  "workflow_id": "abc123",
  "schema_version": "2024.01.15",
  "current_step": "awaiting_approval",
  "data": { ... }
}

# On load: check schema_version
# Match → proceed
# Mismatch → migrate state, or route to old code path

Feature Flags as Rollback

if feature_flags.is_enabled("new_email_system"):
    send_via_new_provider(msg)
else:
    send_via_legacy(msg)  # old path preserved

Not true rollback. Both paths are deployed. But you can disable new behavior without redeploying. Tradeoff: both paths need maintenance until the flag is removed.

Release Artifacts

release-2.4.0/
├── SPEC.md          # Full spec at this version
├── SPEC_DIFF.md     # Diff from previous version
├── MIGRATION/
│   ├── database/    # SQL migrations
│   ├── data/        # Data migration scripts
│   └── state/       # Workflow state migrations
├── ROLLBACK.md      # How to undo (if possible)
└── DEPLOY.md        # Deployment sequence

Migration Risk Levels

TypeExampleRisk
AdditiveNew nullable columnSafe, ship freely
Additive + defaultNew non-null columnSafe if default is sensible
StructuralRelationship changeNeeds expand-migrate-contract
DestructiveColumn removalNeeds approval + archival

Failure Modes

Spec drift: Hotfixes shipped without updating specs. Reality and spec diverge. Fix: validate spec matches implementation before computing diffs. Never ignore drift.

Partial migration: Migration fails halfway. System matches neither old nor new spec. Fix: all migrations must be transactional or resumable. Generate both "continue forward" and "roll back to start" scripts.

Cascade misses: Agent changes component A but misses that B depends on A. Fix: architecture templates capture dependencies. Integration tests exercise boundaries.

Rules for Agents in Evolution Mode

  • Never generate DROP without explicit human instruction
  • Never assume data loss is acceptable
  • Never skip rollback scripts for reversible changes
  • Flag any migration that locks production tables
  • Preservation is the default: anything not in the diff stays unchanged

A project shifts from greenfield to evolution at the first real user, first irreplaceable data, or first live integration. Before that: regenerate freely. After: diffs only.

Hyrum's Law: every observable behavior becomes depended-on behavior. Gall's Law: complex systems that work evolved from simple systems that worked. Both say the same thing: evolve live systems from diffs, not regeneration.
