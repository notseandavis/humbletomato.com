Continuous Evolution

Extending spec-driven development to live products with real users, real data, and real constraints. Work from diffs, not regeneration.

The Problem

The Matrix Methodology works beautifully for greenfield development. Regenerate from specs daily, iterate fast, learn quickly.

But what happens when you have:

  • Paying customers using the system
  • Production data that can't be recreated
  • API contracts with external consumers
  • In-progress business workflows
  • Compliance and audit requirements

The naive answer — "just regenerate everything" — fails immediately. You can't drop production databases. You can't break API contracts. You can't invalidate in-flight workflows.

Work from the diff, not the whole

The spec is still the source of truth, but the agent works from the diff, not the whole.

In greenfield mode, the agent generates from a complete spec. The spec describes the entire system; the agent builds the entire system.

In evolution mode, the agent generates from a spec delta. The canonical spec still describes the entire system, but the agent's work order is the difference between version N and version N+1.

Greenfield:  Spec → Agent → System
Evolution:   Spec(N) → Spec(N+1) → Diff → Agent → Targeted Changes

What Gets Diffed

The spec isn't one document — it's a constellation. Each produces its own diff:

Spec Layer Diff Produces Agent Action
PRD Feature delta New/modified code paths
Data Model Entity changes Migration scripts + code changes
API Contract Endpoint changes Versioned endpoints, compatibility layer
Templates Structure changes Code reorganization (rare, high-risk)
Integrations Connection changes New adapters, deprecated old ones

Database Migrations: The Hard Part

In greenfield, agents generate schemas. In evolution, agents generate migrations — and migrations have properties that schemas don't:

  • Order matters — Migration A must run before B
  • Data preservation — Existing rows must survive
  • Rollback possibility — Some changes are reversible, some aren't
  • Runtime impact — Some migrations lock tables, some don't

Migration Classification

The agent must classify each change:

Type Description Risk
Additive New column, nullable Safe, no data migration
Additive with default New column, non-null Safe if default is sensible
Structural Relationship change Requires data migration script
Destructive Column removal Requires approval + archival

The Three-Phase Migration Pattern

For structural changes, use expand-migrate-contract:

Phase 1: Expand

  • Add new columns/tables alongside old ones
  • Code writes to both old and new
  • Code reads from old (authoritative)

Phase 2: Migrate

  • Backfill script populates new structure from old
  • Validation confirms data integrity
  • Code switches to read from new (authoritative)
  • Code still writes to both

Phase 3: Contract

  • Remove old columns/tables
  • Code only uses new structure
  • Rollback no longer possible — one-way door

This is slow by design. In greenfield, you sprint. In evolution, you walk carefully.

What Agents Must Never Do

  • Never generate DROP without explicit instruction
  • Never assume data loss is acceptable
  • Never generate migrations that lock production tables without flagging it
  • Never skip the rollback script for reversible changes

These constraints belong in architecture templates, enforced by validation.

API Contract Evolution

In greenfield, APIs are generated fresh. In evolution, APIs are promises to consumers. Breaking them breaks trust — and possibly production.

Change Classification

Non-breaking (ship freely):

  • New optional fields in responses
  • New optional parameters in requests
  • New endpoints entirely
  • Performance improvements (same contract, faster)

Breaking (requires versioning):

  • Removed fields from responses
  • New required parameters in requests
  • Changed field types
  • Changed semantics (same field, different meaning)

Versioning Strategies

URL Versioning: /api/v1/users → /api/v2/users

  • Explicit, easy to understand
  • Consumers must update URLs
  • Agent generates new version + deprecation warning on old

Header Versioning: Accept: application/vnd.company.v2+json

  • Same URL, version in headers
  • More elegant, harder to debug
  • Agent generates content negotiation layer

Additive Versioning: Never break, only add

  • New fields are optional, old fields stay forever
  • Simplest consumer experience
  • Creates bloat over time
  • Agent adds new fields, never removes

The Deprecation Pattern

When a breaking change is needed, the agent generates:

  1. New endpoint/version with updated contract
  2. Compatibility layer translating old → new (where possible)
  3. Deprecation notice in old endpoint responses
  4. Sunset timeline in documentation
  5. Migration guide for consumers

The old endpoint doesn't disappear until the sunset date passes, usage metrics show zero traffic (or acceptable loss), and a human approves removal.

Workflow State Management

The Straddler Problem

User starts a multi-step workflow on Monday. You deploy a change on Tuesday. User tries to complete on Wednesday.

Their session state assumes the old world. Your code assumes the new world. Things break.

Versioned State Machines

Every persisted workflow should include its schema version:

{
  "workflow_id": "abc123",
  "schema_version": "2024.01.15",
  "current_step": "awaiting_approval",
  "data": { ... }
}

When code loads a workflow:

  1. Check schema version against current version
  2. If match: proceed normally
  3. If mismatch: run state migration or graceful handling

State Migration Strategies

Option A: Migrate on Load

  • Old workflow loaded
  • Migration function transforms state to new schema
  • Save migrated state, proceed with new code
  • Works when: Transformation is deterministic and lossless

Option B: Dual Code Paths

  • Detect schema version at load
  • Route to old code path or new code path
  • Old workflows complete under old rules
  • Works when: Migration is complex or lossy

Option C: Force Restart

  • Old workflow can't continue
  • User notified, workflow reset to beginning
  • Compensation/apology if appropriate
  • Works when: All else fails (use sparingly)

Spec Versioning

Specs need semantic versioning, but the semantics are different from code:

SPEC_VERSION = MAJOR.MINOR.PATCH

MAJOR: Breaking changes to data model or core concepts
MINOR: New features, non-breaking additions
PATCH: Clarifications, bug fixes in spec language

The spec version isn't the software version — it's the contract version. Multiple software releases might implement the same spec version.

Release Artifacts

Each release should include:

release-2.4.0/
├── SPEC.md                    # Full spec at this version
├── SPEC_DIFF.md               # Diff from previous version
├── MIGRATION/
│   ├── database/              # SQL migrations
│   ├── data/                  # Data migration scripts
│   └── state/                 # Workflow state migrations
├── CHANGELOG.md               # Human-readable changes
├── ROLLBACK.md                # How to undo (if possible)
└── DEPLOY.md                  # Deployment sequence

Rollback Strategy

Rollback is not symmetric with deployment. Data changes make some rollbacks impossible.

Rollback Classification

Level 1: Fully Reversible

  • Rollback = deploy previous version
  • No data changes
  • Agent generates forward and rollback migrations

Level 2: Requires Data Restore

  • Rollback = previous version + restore data from backup
  • Downtime while restore runs
  • Agent generates backup requirements in DEPLOY.md

Level 3: Forward Only

  • Rollback not possible (or cost-prohibitive)
  • Must fix forward
  • Agent flags this explicitly, requires additional approval

Feature Flags as Rollback Substitute

For Level 2 and 3 changes, generate feature flags:

if feature_flags.is_enabled("new_email_system"):
    # New code path
else:
    # Old code path preserved

This isn't rollback — both paths are deployed. But it lets you disable new functionality without redeploying.

The tradeoff: code complexity. Both paths must be maintained until the flag is removed.

Greenfield vs Evolution Mode

The agent needs to know which mode it's operating in. This changes behavior:

Aspect Greenfield Evolution
Input Full spec Spec diff
Optimize for Speed, completeness Safety, precision
Validation Does output match spec? Does output match diff AND preserve existing behavior?
Wrong output Regenerate Might break production
Database Reset and retry Data loss is catastrophic
Speed vs Safety Speed Safety

When Does the Transition Happen?

A project shifts from greenfield to evolution when:

  • First customer is using the system
  • First data exists that can't be recreated
  • First external integration is live

Before this point: regenerate freely. After: diffs only.

New Failure Modes

Evolution introduces failure modes that don't exist in greenfield:

Spec Drift

Problem: Reality diverges from spec. Hotfixes shipped without updating specs. Manual database changes.

Detection: Before computing diffs, validate current spec matches current implementation.

Recovery: Update spec to match reality OR generate changes to make reality match spec. Never ignore drift.

Partial Migration

Problem: Migration runs halfway and fails. System is in a state matching neither old nor new spec.

Prevention: All migrations must be transactional (all-or-nothing) or resumable (can pick up where they left off).

Recovery: Agent generates both "continue forward" and "roll back to start" scripts.

Cascade Misses

Problem: Agent changes component A per the diff but misses that component B depends on A and also needs changes.

Prevention: Architecture templates capture dependencies. Diff computation includes transitive dependencies.

Detection: Integration tests that exercise component boundaries.

Operationalizing Evolution

Minimum Viable Evolution Workflow

  1. Spec lives in version control — Every change is a commit
  2. Diff is computed automatically — CI generates diff from git
  3. Agent prompt includes diff — Standard template for evolution
  4. Migration is first-class — Dedicated section in every change set
  5. Human reviews migrations — Hard gate before deployment
  6. Rollback is documented — Part of release artifact

Tools Agents Need

  • Schema diff tool — Compare data models structurally
  • API diff tool — Compare OpenAPI/GraphQL specs
  • Codebase search — Find all usages of changed elements
  • Test database — Run migrations safely before production
  • Feature flag system — Enable gradual rollout

Principles

  1. The diff is the work order. Don't let the agent wander.
  2. Preservation is the default. Anything not in the diff should remain unchanged.
  3. Migrations are code. They deserve the same review, testing, and versioning.
  4. Breaking changes are expensive. Price accordingly.
  5. Rollback is a first-class feature. Know how to unship before shipping.
  6. Drift is debt. Pay it down regularly.
  7. Agent is conservative; human is decisive. In evolution mode, agents flag risks and wait for decisions.

Summary

Matrix Methodology works for greenfield. Continuous Evolution extends it to live products.

The key insight: work from the diff, not the whole. The canonical spec is still complete, but the agent's work order is the delta between versions.

This requires versioned specs, first-class migration handling, API versioning, state versioning, explicit rollback classification, modified prompts and validation, and awareness of new failure modes.

It's more complex than greenfield. That complexity is necessary — you're managing a living system with real users and real data. The goal isn't to eliminate complexity but to structure it so agents can handle routine parts while humans make decisions that matter.

The agent becomes less of a builder and more of an evolver. Same methodology, different mode.
