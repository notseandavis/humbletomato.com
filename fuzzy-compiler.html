Fuzzy Compiler

An LLM is a compiler that accepts natural language as source code. The compilation is probabilistic.

The Mental Model

A traditional compiler takes precise source code and produces deterministic output. Same input, same binary, every time. The contract is exact: you write valid syntax, you get correct machine code.

An LLM does the same job — takes a specification and produces code — but the contract is fuzzy. The input is natural language. The output is probabilistic. Same prompt, different run, slightly different code.

Traditional compiler:
  source code → deterministic → binary
  Precise input. Exact output. Every time.

Fuzzy compiler:
  natural language spec → probabilistic → source code
  Ambiguous input. Approximate output. Usually.

What Changes

When you accept that you're writing source code for a fuzzy compiler, everything about how you write specs changes.

A traditional compiler rejects ambiguity — it throws a syntax error. A fuzzy compiler resolves ambiguity — it picks an interpretation and keeps going. It never tells you it guessed. It just builds what it thinks you meant.

Traditional compiler:
  "int x = ???"  →  ERROR: expected expression

Fuzzy compiler:
  "handle auth"  →  *builds session-cookie auth*
                     (you wanted JWT)
                     (it didn't ask)

Every ambiguity in your spec is a branch point where the compiler might go a different direction than you intended. The more ambiguity, the more branches, the further the output drifts from what you actually wanted.

Compiler Errors Look Different

Traditional compilers fail loudly — red text, line numbers, clear errors. You fix the syntax, recompile, move on.

Fuzzy compilers fail silently. The code compiles. It runs. It even looks right. But it made assumptions you didn't intend, chose patterns you didn't want, and resolved ambiguity in directions you didn't expect. The "errors" are semantic, not syntactic.

Traditional compiler error:
  Line 42: undefined reference to 'processPayment'
  → Fix: define the function

Fuzzy compiler "error":
  Code works. Tests pass. Ships to production.
  Three weeks later: "Why does checkout fail 
  when the cart has mixed currencies?"
  → The spec never mentioned multi-currency.
  → The compiler assumed single-currency.
  → Nobody noticed.

Writing Better Source Code

If the LLM is your compiler, your spec is your source code. The quality of your spec determines the quality of your compilation — not as a nice-to-have, but as a conservation law.

You wouldn't write half a function and expect the C compiler to figure out the rest. Don't write half a spec and expect the fuzzy compiler to guess correctly.
