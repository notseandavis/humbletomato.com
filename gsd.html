GSD (Get Shit Done)

A meta-prompting and context engineering system that solves quality degradation by giving each task a fresh 200K token context window.

Source: GitHub · gsd.build · Created by Lex Christopherson

The Problem: Context Rot

AI quality degrades as the context window fills. At 0-30% context, output is thorough and comprehensive. Past 50%, the model starts rushing and cutting corners. Past 70%, hallucinations and forgotten requirements appear. Task 50 gets worse output than task 1 from the same model.

GSD's solution: spawn a fresh sub-agent for each task. Every task gets a clean 200K token window with only what it needs. No conversation history baggage.

What It Actually Is

GSD v1 is ~50 markdown files: 29 slash commands, 12 agent definitions, and 2 hooks. No proprietary runtime. Everything runs on native Claude Code features. A Node.js CLI helper handles deterministic logic (file existence checks, config loading) so the LLM focuses on reasoning, not bookkeeping.

GSD v2 is a standalone TypeScript CLI that programmatically controls agent sessions instead of relying on the LLM to follow prompt instructions. It adds crash recovery, stuck detection, cost tracking, and git worktree isolation.

The Four-Command Workflow

/gsd:new-project    # Deep interview + parallel research agents
/gsd:discuss-phase  # Capture preferences for gray areas
/gsd:build          # Atomic plans, wave parallelism, fresh agents
/gsd:verify         # Goal-backward: "is it TRUE?" not "was it done?"

New Project

Interviews you about what you're building while spawning four parallel research agents: stack, features, architecture, and pitfalls. Each investigates independently, then a synthesizer merges their findings.

Outputs: PROJECT.md, REQUIREMENTS.md, ROADMAP.md, STATE.md

Discuss Phase

The most underrated feature. Before planning, GSD identifies gray areas specific to what you're building (layout choices for visual features, error handling for APIs, structure for content systems) and asks targeted questions. Your answers go into CONTEXT.md and every subsequent agent gets them.

Skipping this is technically optional but is the #1 source of misaligned output. Plans inherit assumptions that CONTEXT.md would have prevented.

Build

Creates atomic PLAN.md files, each containing 2-3 tasks that fit in ~50% of a context window. Plans run in waves: independent tasks run in parallel, dependent tasks wait for their wave.

Plans are prompts. Not documents that become prompts. The sub-agent reads PLAN.md directly as its executable instruction. No summarization or interpretation layer. Plan quality directly determines execution quality.

Plans use XML structure optimized for Claude's training:

<task type="auto">
  <name>Create login endpoint</name>
  <files>src/app/api/auth/login/route.ts</files>
  <action>Use jose for JWT. Validate against users table.</action>
  <verify>curl POST localhost:3000/api/auth/login returns 200</verify>
</task>

Each plan produces an atomic git commit. Each task gets its own clean agent session.

Verify

Goal-backward verification checks whether goals are TRUE, not whether tasks were completed. "Can a user sign up with email?" not "Does the signup route exist?" If verification fails, new plans are created rather than retrying the same tasks.

Strengths

StrengthDetail
Context engineeringFresh 200K windows per task. No degradation from first to last.
Simple workflowFour commands. Complexity is in the system, not the interface.
Wave parallelismIndependent tasks run simultaneously. Works best with vertical slice decomposition.
Plan checkerPlans pass verification (up to 3 iterations) before execution. Checks scope, dependencies, conflicts.
Honest creatorv2 README openly acknowledges v1's limitations. No context control, no crash recovery, no observability.

Tradeoffs

LimitationDetail
Token hungryFull workflow burns through Pro plan limits fast. Multi-agent spawning is expensive.
Greenfield biasLarge brownfield codebases with deep dependency chains can exceed what a single task comprehends.
Planning quality variesQuality depends on the discuss phase. Skip it and plans inherit bad assumptions.
Solo-orientedLess structure for multi-person coordination than frameworks like BMAD.
MCP incompatibilityCustom sub-agents can't use MCP servers (Claude Code limitation, not GSD's fault).

When to Use GSD

Solo or small team. Greenfield or small-to-medium project. You want agents to handle implementation while you focus on product decisions. You value simplicity over comprehensive structure.

When to Look Elsewhere

Large brownfield codebases. Tightly coupled systems that don't decompose into independent tasks. Teams needing role separation and audit trails. Quick fixes (though /gsd:quick exists for ad-hoc tasks).
