GSD (Get Shit Done)

Context rot solved. Fresh sub-agent contexts per task mean task 50 has the same quality as task 1.

Source: GitHub (11.9K+ stars) · gsd.build · Created by Lex Christopherson

The Core Problem: Context Rot

Quality degrades as context fills. Task 1 gets clean instructions and 200K tokens to work with. Task 50 gets instructions buried under 49 previous conversations and maybe 20K tokens left.

Same agent, worse results. Not because the model got dumber, but because the signal-to-noise ratio collapsed.

The Solution: Fresh Contexts

Each task spawns a sub-agent with a clean 200K token window. Task 50 gets the same pristine environment as task 1.

The trick: each sub-agent gets exactly what it needs (task definition, architecture context, relevant specs) without dragging along the entire chat history.

How It Works

Four commands. That's it.

/gsd:new-project
/gsd:discuss-phase
/gsd:build
/gsd:verify

1. /gsd:new-project

Interviews you about what you're building. Not surface-level questions. Deep ones.

Spawns parallel research agents to investigate technical choices. While you're explaining your idea, background agents are checking framework docs, API patterns, deployment options.

Outputs four files:

  • PROJECT.md — The what and why
  • REQUIREMENTS.md — What success looks like
  • ROADMAP.md — Phases, not tasks
  • STATE.md — Current status (starts empty)

2. /gsd:discuss-phase

Before planning, capture preferences for gray areas.

Example: "For error handling, do you want fail-fast or graceful degradation?"

These decisions go into context. Agents won't ask again. No more "I'll use approach X unless you prefer Y" waffling in every task.

3. /gsd:build

Creates atomic PLAN.md files. Each plan:

  • Contains 2-3 tasks max
  • Fits in ~50% of context window
  • Gets a fresh sub-agent
  • Produces atomic git commits

Plans run in waves. Wave 1 tasks can run in parallel (no dependencies). Wave 2 waits for Wave 1 completion. And so on.

The parallelism is real. If you have 6 tasks in a wave and processing power to spare, all 6 run simultaneously.

4. /gsd:verify

Goal-backward verification. Not "did you do the tasks?" but "is the goal true?"

Example goal: "Users can sign up with email and password."

Bad verification: Check that signup route exists, password hashing is implemented, database table created.

Good verification: Actually run the signup flow. Create a user. Verify the user can log in.

If verification fails, re-plan. Don't just retry the same tasks.

Plans ARE Prompts

This is subtle but important: PLAN.md isn't a document that gets turned into a prompt. PLAN.md is the prompt.

The sub-agent reads PLAN.md directly. No summarization layer. No interpretation step. What you write in the plan is what the agent executes.

This means:

  • Be specific in plans
  • Include code examples if needed
  • Reference exact file paths
  • State acceptance criteria clearly

The plan quality directly determines execution quality.

State Tracking

STATE.md updates after every task. Contains:

  • What's built
  • What's in progress
  • What's blocked
  • Open questions

When a sub-agent spawns, it gets STATE.md in its context. It knows what already exists. It doesn't re-implement things or make conflicting changes.

Philosophy

"I don't write code. Claude Code does."

Lex calls this anti-enterprise. Not in the sense of anti-corporate, but anti-complexity-in-workflow. The workflow is four commands. The complexity is hidden inside the system.

This is for solo builders. People who want to ship, not manage processes.

Compatible Tools

Works with Claude Code, OpenCode, Gemini CLI, Codex, Copilot, Antigravity. Anything that can spawn sub-agents and work from markdown prompts.

What It's Good At

Strength Why
Context management Fresh windows per task, no degradation
Simple UX Four commands, clean mental model
Solo builders Battle-tested by people shipping real products alone
Speed Wave-based parallelism when dependencies allow

Tradeoffs

Limitation Impact
Team coordination Less structure for multi-person projects
Planning depth Quality depends on interview responses
Tooling opinions Optimized for specific CLI tools
Upfront thinking Requires good answers during new-project phase

When to Use GSD

You're building solo or with a small team. You want agents to handle implementation while you focus on product decisions. You value simplicity in workflow over comprehensive structure.

You're comfortable with the tools it integrates with. You want something that works out of the box without configuration.

When to Look Elsewhere

You need heavyweight project management structure. You have a large team that needs role separation. You want extensive templates and workflows for every phase. You need support for non-technical stakeholders in the loop.

In those cases, consider BMAD or similar methods that trade simplicity for comprehensive lifecycle coverage.
