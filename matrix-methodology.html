Matrix Methodology

Three inputs. One context window. Implementation resolves like a diffusion model.

The Context Matrix

Matrix is a framework for composing the context window. It assumes you can one-shot any task if the context is right. The question: what inputs produce predictable code?

Three inputs form a literal matrix:

                          Architecture (columns)
                    ┌───────────┬───────────┬───────────┐
                    │   Auth    │    API    │   State   │
         ┌──────────┼───────────┼───────────┼───────────┤
  PRD    │  Login   │  OAuth2 + │  POST     │  Session  │
 (rows)  │          │  JWT      │  /login   │  store    │
         ├──────────┼───────────┼───────────┼───────────┤
         │  Cart    │  User     │  CRUD     │  Redis    │
         │          │  scoped   │  /cart/*  │  cache    │
         ├──────────┼───────────┼───────────┼───────────┤
         │  Payment │  PCI      │  POST     │  Tx log   │
         │          │  tokenize │  /pay     │  + saga   │
         └──────────┴───────────┴───────────┴───────────┘

Each cell is a resolution point. PRD row (what the business wants) meets architecture column (how this platform builds things). The cell contains the resolved implementation decision.

The spec template is the function that fills each cell. It takes (business requirement, technical pattern) and outputs the specific implementation choice. "Login" + "Auth patterns" = "OAuth2 with JWT rotation." "Cart" + "State management" = "Redis-cached, user-scoped."

Simultaneous Resolution

Here's where it gets interesting. In a diffusion model, the entire image resolves simultaneously. The eyes know about the nose. The background knows about the foreground. Every pixel is constrained by every other pixel in the same denoising step.

The context matrix works the same way. All three inputs sit in one context window. When the agent resolves "Login + Auth," it already sees "Payment + Auth" in the same context. The auth decisions are consistent across the entire matrix because they resolve together, not in isolation.

Diffusion model:    noise  →  →  →  coherent image
                    (all pixels resolve together)

Context matrix:     PRD + Architecture + Spec Template  →  coherent implementation
                    (all resolution points resolve together)

This is why one-shot works when the context is right. The agent isn't resolving cells sequentially. It's resolving the entire implementation surface at once, with every decision constrained by every other decision in the same window.

The Spec Template as Resolution Function

The spec template maps abstract business intent to concrete implementation. It resolves each cell:

# Spec Template (abbreviated)

## Authentication
- Strategy: OAuth2 + JWT (from Architecture/auth-patterns.md)
- Login flow: POST /api/auth/login → JWT pair
- Session: Redis-backed, 15min access / 7d refresh

## Cart  
- Scope: User-bound via JWT subject claim
- Storage: Redis cache (hot) + Postgres (durable)
- API: CRUD at /api/cart/* with optimistic locking

## Payment
- Tokenization: Stripe tokenization (never raw card data)
- Flow: POST /api/payments → saga with compensating transactions
- Audit: Immutable transaction log, append-only

Each section is a row of resolved cells. The spec template adds information that didn't exist in either source: the specific resolution of abstract intent into concrete decisions. Writing it forces you to see where the PRD and architecture don't connect. If a cell doesn't resolve cleanly, you've caught a problem before any code is generated.

The PRD is also fed directly to the agent alongside the spec. The agent sees both raw business intent and the resolved mapping. Maximum information, minimum loss (see The Telephone Game).

Templates Compound

Architecture templates control the shape of each column. Better auth template? Every row that touches auth improves. Better API conventions? Every endpoint benefits. The 10th feature generates faster and cleaner than the 1st because the column patterns have been refined by the previous nine.

What Matrix Doesn't Cover

Matrix composes the context. It doesn't prescribe what happens when a task exceeds one window (that's the Ralph Loop), how to coordinate multiple agents (left to the practitioner), or how to evolve live systems (see Continuous Evolution).

For how this changes daily development practice, see Matrix Way of Working.

Strengths

StrengthDetail
Simultaneous resolutionAll decisions resolve in one context window. Auth knows about state knows about API. Consistency emerges from shared context, not coordination.
Template compoundingBetter column patterns improve every future row. Investment in architecture templates pays compound interest.
Information-theoretic foundationRaw PRD + architecture fed directly to agent avoids lossy intermediate synthesis. Fewer transformations = less signal loss.
Validation checkpointWriting the spec template surfaces gaps before code generation. If a cell doesn't resolve, the inputs don't connect.

Tradeoffs

LimitationDetail
Template investmentColumn quality determines output quality. Building good architecture templates requires deep platform knowledge and significant upfront effort.
Context window ceilingPRD + architecture + spec must fit in one window. Large matrices with many rows and columns hit this limit. Complex enterprise systems may not fit.
"Deterministic" is aspirationalLLM output is probabilistic. Constrained solution spaces reduce variance but don't eliminate it. Same inputs can produce different resolutions across runs.
No built-in orchestrationUnlike GSD's wave parallelism or BMAD's agent personas, Matrix doesn't prescribe multi-agent coordination.
Spec writing is a skillFilling the matrix requires someone who understands both the business rows and the architecture columns. That person is rare.

Compare

GSD focuses on context engineering per-task with fresh sub-agent windows. BMAD uses agent personas to simulate team roles across a structured lifecycle. Matrix focuses on composing the single context window that makes one-shot resolution possible.
