Matrix Methodology

Spec + Architecture = Implementation

Spec + Architecture = Implementation

That's the whole idea. The spec says what to build. The architecture says how this platform builds things. Where they intersect, the implementation resolves.

                        Architecture
                  ┌──────────┬──────────┬──────────┐
                  │   Auth   │   API    │  State   │
       ┌──────────┼──────────┼──────────┼──────────┤
       │  Login   │ OAuth2 + │ POST     │ Redis    │
 Spec  │          │ JWT      │ /login   │ sessions │
       ├──────────┼──────────┼──────────┼──────────┤
       │  Cart    │ User-    │ CRUD     │ Redis +  │
       │          │ scoped   │ /cart/*  │ Postgres │
       ├──────────┼──────────┼──────────┼──────────┤
       │  Payment │ PCI      │ POST     │ Tx log   │
       │          │ tokenize │ /pay     │ + saga   │
       └──────────┴──────────┴──────────┴──────────┘

Each cell is a resolution point. Remove either side and the agent guesses:

Spec alone (no architecture)
"Build login, cart, and checkout" — Agent invents its own auth model, state strategy, API style.

Architecture alone (no spec)
"We use OAuth2, Redis, REST conventions" — Agent has no idea what to build.

Both
"Login with OAuth2+JWT, cart user-scoped via JWT subject, checkout via Stripe tokenization with saga pattern" — Implementation resolves.

Business Requirements Sit Alongside

The raw business context isn't in the matrix. It sits next to it. The agent sees both the resolved spec and the original intent, so it understands why each decision was made without re-deriving what the spec already resolves.

Templates Compound

Architecture is the columns. Better auth template? Every row that touches auth improves. Better API conventions? Every endpoint benefits. The 10th feature generates faster than the 1st because the column patterns were refined by the previous nine.

The Spec Surfaces Gaps

Writing the spec forces you to resolve every cell. If "Payment + State" doesn't resolve cleanly, you've found a gap between the business requirements and the architecture before any code is generated. The spec is a validation checkpoint.

Don't Scope the Context

Engineers instinctively isolate. "I'm building the cart service, so I'll scope the context to cart stuff. Less noise." Good instinct for writing code. Wrong instinct for composing context.

The agent building the cart needs to see the auth model (whose cart?), the payment flow (checkout handoff), the inventory rules (stock validation). Not because it's building those things, but because cart decisions are shaped by those things.

Scoped context: "Build the cart service. Here's the cart spec." Cart works in isolation. Breaks the moment it touches auth, payments, inventory.

Full system context: "Build the cart service. Here's the cart spec, the auth model, the payment flow, the inventory rules." Cart integrates correctly on first generation.

Scoping context is just abstraction applied to the context window. It fails for the same reason: you're hiding information the agent will eventually need.

What Matrix Doesn't Cover

Matrix organizes context. It doesn't prescribe execution patterns (Ralph Loop), multi-agent coordination, or how to evolve live systems (Continuous Evolution).

Tradeoffs

  • Template investment. Column quality determines output quality. Good architecture templates require deep platform knowledge.
  • Spec writing is a skill. Filling the matrix requires someone who understands both the business rows and the architecture columns.
  • LLMs are probabilistic. Constrained solution spaces reduce variance but don't eliminate it.
  • No built-in orchestration. Unlike GSD or BMAD, Matrix doesn't prescribe how agents coordinate.
