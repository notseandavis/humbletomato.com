Matrix Way of Working

How the Context Matrix changes daily development practice. Daily releases, evolving specs, and new roles.

Daily Iteration, Not Quarterly Releases

Traditional development: Product thinks (2 weeks) → PRD (1 week) → Architecture review (3 days) → Dev builds (3 weeks) → QA tests (1 week) → Feedback (too late).

With Matrix: daily releases.

DayPRDReleaseResult
1v0.1R0Feedback surfaces gaps
2v0.2R1Feedback surfaces edge cases
3v0.3R2E2E mostly passing → RC ready
4v0.4RC0E2E green, product reviewing
5-RC0 → FINALShip it

Rejection is cheap. The agent rebuilds from updated specs in hours, not weeks. No sunk cost psychology. Nobody spent three weeks on R1, so nobody fights to ship it despite known issues.

The PRD and the code converge together. Neither is "done" first. Both are done when E2E passes.

E2E Tests as Definition of Done

Every cycle has one measure: end-to-end integration tests pass. Not "code looks right." Not "agent completed all tasks." The tests pass or they don't.

The spec template defines expected behavior. E2E tests encode that behavior as assertions. The agent's job isn't "build the feature." It's "make the tests green." Verifiable, no ambiguity.

Spec Quality Levels

Specs climb through maturity levels. You don't need L3 to start.

LevelDescriptionAgent Can...
L0: SketchBullet points, intent onlyGenerate scaffolding, identify questions
L1: DraftStructured sections, some gapsGenerate compilable code, surface missing pieces
L2: SpecifiedComplete sections, edge cases notedGenerate passing unit tests
L3: VerifiedValidated by successful runsGenerate passing E2E tests

Start at L0 and let R0 tell you what's missing. Rapid iteration reveals truth faster than exhaustive upfront planning.

Overcoming Failures

Agents aren't brittle. CLI execution + iterative loops let them self-correct:

SnagHow Agent Overcomes
Build failsReads error, fixes code, rebuilds
Test failsReads assertion, adjusts implementation
Service won't startChecks logs, fixes configuration
Missing dependencySearches reference implementation, adds it

Role Evolution

Product Managers Become Builders

When agents generate code from specs, the person writing the spec is the builder. PMs move from describing what they want to directly shaping what gets built. Their product judgment becomes the primary input to the build loop.

Engineers Become Architects

Engineers shift from writing code to designing the systems that generate it. They own the templates, architecture docs, and infrastructure that make the build loop reliable. They debug the loop, not the code.

Principles

  1. Ship specs daily, not quarterly. Treat PRDs like code: commit, iterate, version.
  2. Let the code challenge the spec. A failing test is a spec review.
  3. Specs and code converge together. Both are done when E2E passes.
  4. Invest in templates, not documents. Better templates improve every future capability.
  5. Start rough, sharpen through execution. An L0 spec that runs today teaches more than an L3 spec next month.
  6. Product stays in the loop daily. Not weekly standups. Daily review of running software.

What This Is Not

Not waterfall with faster typing. Speed is wasted if inputs don't iterate at the same pace.

Not "AI replaces product." Product judgment is more important, not less. Someone has to decide what the error message should say.

Not "AI replaces architecture." Templates and architecture docs are the machine. Engineers keep the machine running.
