The Ralph Wiggum Loop

An autonomous execution pattern for AI agents. Small, focused iterations with fresh context prevent degradation and enable reliable autonomous work.

The Problem Ralph Solves

Real features don't fit in one context window. A login system needs schema changes, middleware, UI components, and session handling. No single prompt covers all of that well. And even if you could fit it, quality degrades as the context fills up.

Ralph breaks work into tasks that each fit in one context window, then iterates with fresh context until the feature is complete.

The Core Pattern

Read fix_plan → Do ONE task → Verify → Update → Report → Repeat

Each iteration is small, focused, and verifiable. The agent commits verified changes rather than generating monolithically. When something breaks, the new issue becomes a blocker in the next loop.

The critical insight: each iteration starts with fresh context. Every loop begins clean with just the specs, architecture docs, and current task. No accumulated cruft from previous iterations. This avoids the context degradation that kills long "vibe coding" sessions.

The Execution Flow

1. Read the PRD

The agent starts by reading prd.json — a structured JSON document with user stories, acceptance criteria, and priorities.

2. Pick the Highest Priority Task

The agent scans for the highest priority user story where passes: false. This is the current task. Only one task per iteration.

3. Implement the Task

The agent writes code, modifies files, and makes the minimum changes needed to satisfy the acceptance criteria for that one story.

4. Run Quality Checks

The agent runs typecheck, lint, tests — whatever the project requires. All checks must pass before committing.

5. Commit Changes

If checks pass, commit ALL changes with a structured message:

feat: [Story ID] - [Story Title]

6. Update the PRD

Set passes: true for the completed story in prd.json.

7. Log Progress

Append to progress.txt with what was implemented, files changed, and learnings for future iterations.

8. Repeat or Complete

If all stories have passes: true, the agent reports completion. Otherwise, another iteration picks up the next task.

PRD-to-JSON Conversion

Before Ralph can run, a product specification must be converted into the structured prd.json format. This conversion is critical — it's where loose product requirements become executable work orders.

Story Sizing: The Number One Rule

Each story must be completable in ONE iteration (one context window).

Ralph spawns a fresh instance per iteration with no memory of previous work. If a story is too big, the model runs out of context before finishing and produces broken code.

Right-sized stories:

  • Add a database column and migration
  • Add a UI component to an existing page
  • Update a server action with new logic
  • Add a filter dropdown to a list

Too big (split these):

  • "Build the entire dashboard" → Split into: schema, queries, UI components, filters
  • "Add authentication" → Split into: schema, middleware, login UI, session handling
  • "Refactor the API" → Split into one story per endpoint or pattern

Rule of thumb: If you cannot describe the change in 2-3 sentences, it's too big.

Story Ordering: Dependencies First

Stories execute in priority order. Earlier stories must not depend on later ones.

Correct order:

  1. Schema/database changes (migrations)
  2. Server actions / backend logic
  3. UI components that use the backend
  4. Dashboard/summary views that aggregate data

Wrong order:

  1. UI component (depends on schema that doesn't exist yet)
  2. Schema change

Acceptance Criteria: Must Be Verifiable

Each criterion must be something Ralph can CHECK, not something vague.

Good criteria (verifiable):

  • "Add status column to tasks table with default 'pending'"
  • "Filter dropdown has options: All, Active, Completed"
  • "Clicking delete shows confirmation dialog"
  • "Typecheck passes"

Bad criteria (vague):

  • "Works correctly"
  • "User can do X easily"
  • "Good UX"
  • "Handles edge cases"

Always include: "Typecheck passes" as a final criterion for every story.

The Progress Log

After each iteration, Ralph appends to progress.txt with:

## [Date/Time] - [Story ID]
- What was implemented
- Files changed
- **Learnings for future iterations:**
  - Patterns discovered
  - Gotchas encountered
  - Useful context
---

The learnings section is critical. It helps future iterations avoid repeating mistakes and understand the codebase better.

Consolidating Patterns

When Ralph discovers a reusable pattern, it gets added to a ## Codebase Patterns section at the TOP of progress.txt:

## Codebase Patterns
- Use sql<number> template for aggregations
- Always use IF NOT EXISTS for migrations
- Export types from actions.ts for UI components

Only add patterns that are general and reusable, not story-specific details. Future iterations read this section first.

Browser Testing Integration

For stories that change UI, verification isn't complete until visually confirmed. Ralph can use browser automation tools to:

  1. Navigate to the relevant page
  2. Verify UI changes work as expected
  3. Take screenshots for the progress log

UI stories include "Verify in browser using dev-browser skill" in their acceptance criteria.

Quality Requirements

Non-negotiable rules:

  • ALL commits must pass quality checks (typecheck, lint, test)
  • Do NOT commit broken code
  • Keep changes focused and minimal
  • Follow existing code patterns

These constraints keep the codebase healthy and prevent compounding technical debt.

Example: Task Status Feature

Here's how a feature request gets converted into Ralph's format:

Original PRD (Markdown)

# Task Status Feature

Add ability to mark tasks with different statuses.

## Requirements
- Toggle between pending/in-progress/done on task list
- Filter list by status
- Show status badge on each task
- Persist status in database

Converted to prd.json

{
  "project": "TaskApp",
  "branchName": "ralph/task-status",
  "description": "Task Status Feature - Track task progress",
  "userStories": [
    {
      "id": "US-001",
      "title": "Add status field to tasks table",
      "description": "Store task status in database",
      "acceptanceCriteria": [
        "Add status column: 'pending'|'in_progress'|'done'",
        "Generate and run migration successfully",
        "Typecheck passes"
      ],
      "priority": 1,
      "passes": false
    },
    {
      "id": "US-002",
      "title": "Display status badge on task cards",
      "description": "Show status at a glance",
      "acceptanceCriteria": [
        "Each task card shows colored status badge",
        "Badge colors: gray=pending, blue=in_progress, green=done",
        "Typecheck passes",
        "Verify in browser"
      ],
      "priority": 2,
      "passes": false
    },
    {
      "id": "US-003",
      "title": "Add status toggle to task list rows",
      "description": "Change task status directly from list",
      "acceptanceCriteria": [
        "Each row has status dropdown or toggle",
        "Changing status saves immediately",
        "UI updates without page refresh",
        "Typecheck passes",
        "Verify in browser"
      ],
      "priority": 3,
      "passes": false
    },
    {
      "id": "US-004",
      "title": "Filter tasks by status",
      "description": "Filter list to see only certain statuses",
      "acceptanceCriteria": [
        "Filter dropdown: All|Pending|In Progress|Done",
        "Filter persists in URL params",
        "Typecheck passes",
        "Verify in browser"
      ],
      "priority": 4,
      "passes": false
    }
  ]
}

Notice how one feature becomes four focused stories, each completable in one iteration, ordered by dependency.

When Ralph Completes

After completing a user story, Ralph checks if ALL stories have passes: true. If so, it replies with:

<promise>COMPLETE</promise>

This signals that the entire feature is done. All acceptance criteria met, all quality checks passed.

Why It Works

The Ralph Loop succeeds where other autonomous patterns fail because:

  • Small iterations keep context manageable and failures contained
  • Fresh context prevents degradation across iterations
  • Verification gates catch issues immediately, not at the end
  • Structured work orders (prd.json) eliminate ambiguity
  • Progress logging builds institutional knowledge
  • Quality requirements prevent broken code from spreading

The pattern is simple enough to implement anywhere but structured enough to produce reliable results.

Use It With Anything

Ralph is a standalone execution pattern. It works with any spec-driven approach: feed it a PRD from BMAD, plans from GSD, or specs from any other framework. All it needs is a structured task list and clear acceptance criteria.
