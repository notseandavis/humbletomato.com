The Solution Space

Vague descriptions produce multiple valid answers. Add constraints and the possibilities shrink. This is why specs matter.

Spec: "Sort a list"

Three valid solutions:

# Solution A: built-in
def sort_list(items):
    return sorted(items)

# Solution B: bubble sort
def sort_list(items):
    n = len(items)
    for i in range(n):
        for j in range(0, n-i-1):
            if items[j] > items[j+1]:
                items[j], items[j+1] = items[j+1], items[j]
    return items

# Solution C: quicksort
def sort_list(items):
    if len(items) <= 1:
        return items
    pivot = items[len(items) // 2]
    left = [x for x in items if x < pivot]
    middle = [x for x in items if x == pivot]
    right = [x for x in items if x > pivot]
    return sort_list(left) + middle + sort_list(right)

All correct. The spec didn't say which one.

Add constraint: "In-place, O(n log n)"

A is out (not in-place). B is out (O(n²)). C is out (creates new lists). You're left with heapsort or in-place mergesort.

Add constraint: "Stable"

Heapsort is out. Only in-place mergesort remains. Each constraint eliminated implementations.

Spec: "Fetch user data from API"

// Solution A: fetch
async function getUser(id) {
    const response = await fetch(`/api/users/${id}`);
    return response.json();
}

// Solution B: axios
async function getUser(id) {
    const response = await axios.get(`/api/users/${id}`);
    return response.data;
}

// Solution C: XMLHttpRequest
function getUser(id, callback) {
    const xhr = new XMLHttpRequest();
    xhr.open('GET', `/api/users/${id}`);
    xhr.onload = () => callback(JSON.parse(xhr.responseText));
    xhr.send();
}

Different libraries, different patterns. All fetch data.

Add constraint: "Retry 3x with exponential backoff"

async function getUser(id) {
    const maxRetries = 3;
    let attempt = 0;
    
    while (attempt < maxRetries) {
        try {
            const response = await fetch(`/api/users/${id}`);
            if (!response.ok) throw new Error(`HTTP ${response.status}`);
            return response.json();
        } catch (error) {
            attempt++;
            if (attempt >= maxRetries) return null;
            await new Promise(r => setTimeout(r, Math.pow(2, attempt) * 1000));
        }
    }
}

The solution space collapsed. There's basically one way to do this. You can bikeshed the details, but the structure is set.

Spec: "Store user theme preference"

// A: localStorage
localStorage.setItem('theme', theme);

// B: cookie
document.cookie = `theme=${theme}; max-age=31536000`;

// C: server
await fetch('/api/preferences', { method: 'POST', body: JSON.stringify({ theme }) });

// D: file system
fs.writeFileSync('user-prefs.json', JSON.stringify({ theme }));

Add constraint: "Persist across devices"

A, B, D are out. Only server-side storage works.

Add constraint: "Real-time sync without polling"

Now you need WebSockets or SSE. REST won't cut it. The constraint eliminated entire categories of solutions.

Why This Matters

When you tell an AI to build something, every detail you omit expands the solution space. The AI picks one valid answer, but you might have expected a different one.

This isn't the AI being stupid. It's math. A system with fewer equations than unknowns has infinite solutions. A spec with fewer constraints than decisions has multiple valid implementations.

Vague spec → wide solution space → surprising outputs
Precise spec → narrow solution space → predictable outputs

Spec-driven development works not because it makes AI smarter, but because it eliminates degrees of freedom. Be intentional about which parts you nail down and which you leave open.
