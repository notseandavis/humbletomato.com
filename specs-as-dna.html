Specs as DNA

Why specifications are genetic code, agents are expression machinery, and intermediate synthesis destroys information.

The DNA Analogy

Specifications are like DNA. The AI agent is gene expression machinery.

DNA doesn't build an organism directly. It encodes instructions that, when expressed in the right cellular environment, produce proteins that assemble into structures. The DNA itself is inert. It's the machinery of gene expression that turns code into life.

Specs work the same way. A PRD, data model, and API contract sitting in a repo don't do anything on their own. But when an AI agent "expresses" them — reading the specs, interpreting them against architectural patterns, and executing in a live development environment — working software emerges.

The implication

If you get the DNA right — clean, complete specifications — the expression is almost deterministic. Bad specs produce malformed output, just like mutations produce malformed proteins.

The methodology's emphasis on spec quality isn't bureaucratic overhead. It's genetic hygiene.

The Diffusion Model Analogy

The iterative generation process works like a diffusion model resolving an image.

A diffusion model starts with noise and progressively refines it into a coherent picture, guided by the prompt and the model's learned understanding of structure.

The agent starts with an empty project — noise — and iteratively resolves it into working software. The specifications are the prompt. Architecture patterns are the learned structure. Each iteration of the execution loop is a denoising step.

Why Raw Inputs Beat Synthesized Ones

A natural instinct: take your PRD and architecture docs, synthesize them into one clean "spec" document, then feed that to the agent. This is a mistake.

The Data Processing Inequality from information theory explains why. For any processing chain X → Y → Z:

I(X; Z) ≤ I(X; Y)

Processing can only preserve or destroy information. It cannot create it. When you synthesize documents X and Y into document Z, you haven't added information. You've compressed it — lossily.

What gets lost

  • Redundancy — The same concept described two different ways in X and Y gives the agent two chances to understand it. Z picks one description and discards the other.
  • Context — The way information is structured in the original document carries meaning. Reorganizing it loses structural context.
  • Edge cases — Synthesis tends to capture the main points and drop the details. The details are where bugs live.
  • Implicit information — What's adjacent to what in the original documents encodes relationships. Synthesis disrupts these.

When synthesis IS valuable

The exception: when the synthesis step adds genuinely new information that wasn't in either source document.

  • Human judgment — Resolving ambiguities between X and Y that the agent can't resolve alone
  • Derived constraints — "Given the data model AND the API contract, the batch size must be < 1000" (neither doc says this alone)
  • Conflict resolution — When X and Y contradict each other, a human choosing which is correct adds information

If your synthesis step is just reformatting or combining, skip it. Feed the raw documents. The agent handles multiple inputs better than you think.
