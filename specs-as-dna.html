Specs as DNA

Why specifications are genetic code, agents are expression machinery, and intermediate synthesis destroys information.

The DNA Analogy

Specifications are like DNA. The AI agent is gene expression machinery.

DNA doesn't build an organism directly. It encodes instructions that, when expressed in the right cellular environment, produce proteins that assemble into structures. The DNA itself is inert. It's the machinery of gene expression that turns code into life.

Specs work the same way. A PRD, data model, and API contract sitting in a repo don't do anything on their own. But when an AI agent "expresses" them — reading the specs, interpreting them against architectural patterns, and executing in a live development environment — working software emerges.

Example: Same Spec, Different Expression

Here's a simple spec (the "DNA"):

# User Story: Display User Profile
- Show user's name, email, join date
- Format join date as "Member since MMM YYYY"
- If user has no email, show "No email on file"

Express this spec with React architecture context:

function UserProfile({ user }) {
  const joinDate = new Date(user.joinedAt)
    .toLocaleDateString('en-US', { 
      month: 'short', 
      year: 'numeric' 
    });

  return (
    <div className="profile">
      <h2>{user.name}</h2>
      <p>{user.email || "No email on file"}</p>
      <p>Member since {joinDate}</p>
    </div>
  );
}

Express the same spec with Vue architecture context:

<template>
  <div class="profile">
    <h2>{{ user.name }}</h2>
    <p>{{ user.email || "No email on file" }}</p>
    <p>Member since {{ formattedJoinDate }}</p>
  </div>
</template>

<script>
export default {
  props: ['user'],
  computed: {
    formattedJoinDate() {
      return new Date(this.user.joinedAt)
        .toLocaleDateString('en-US', { 
          month: 'short', 
          year: 'numeric' 
        });
    }
  }
}
</script>

Same spec ("DNA"), different expression ("proteins"). The spec is framework-agnostic. The architecture context determines the implementation.

Change the spec, output changes predictably:

# User Story: Display User Profile
- Show user's name, email, join date
- Format join date as "Member since MMM YYYY"
- If user has no email, show "No email on file"
+ If user is premium, show gold star badge next to name

React output now includes:

<h2>
  {user.name}
  {user.isPremium && <span className="badge">⭐</span>}
</h2>

The spec mutation propagates to the code. Just like a DNA mutation changes protein structure.

How to Write Good Specs

Not all specs constrain the solution space effectively. Here's the difference:

Bad Spec (Too Vague)

# Feature: User Settings
Users need to be able to change their preferences.
Make it easy to use.

This produces unpredictable output. What preferences? What UI pattern? How is data stored?

Better Spec (Constrained)

# Feature: User Settings

## Data Model
- Add `user_preferences` table
  - user_id (FK to users)
  - theme: 'light' | 'dark' | 'auto'
  - email_notifications: boolean
  - language: string (ISO 639-1)

## UI Requirements
- Settings page at /account/settings
- Form with labeled inputs for each preference
- "Save Changes" button (disabled until form is dirty)
- Show "Saved!" confirmation for 2 seconds on success
- Validate: language must be one of ['en', 'es', 'fr', 'de']

## API Contract
POST /api/user/preferences
Body: { theme, email_notifications, language }
Returns: 200 + updated preferences object
         400 + validation errors if invalid

## Acceptance Criteria
- Changing theme updates UI immediately (no page reload)
- Invalid language shows inline error
- Closing tab without saving shows "unsaved changes" warning
- Typecheck passes

Now the agent knows exactly what to build. The solution space is narrow.

Pattern: Constrain Without Over-Specifying

Good specs define what and constraints, not how at the code level:

Too VagueJust RightOver-Specified
"Make it fast" "Response time <200ms p95" "Use Redis with 60s TTL and read-through caching pattern"
"Handle errors" "Show user-friendly message on network failure, log technical details" "Use try-catch with console.error() and toast notification via react-hot-toast"
"Users can search" "Full-text search, debounced 300ms, max 20 results" "Use Elasticsearch with ngram tokenizer and custom scoring function"

The "Just Right" column lets the agent choose implementation details based on architecture context. The "Over-Specified" column locks in decisions that might be wrong for the framework.

The implication

If you get the DNA right — clean, complete specifications — the expression is almost deterministic. Bad specs produce malformed output, just like mutations produce malformed proteins.

The methodology's emphasis on spec quality isn't bureaucratic overhead. It's genetic hygiene.

The Diffusion Model Analogy

The iterative generation process works like a diffusion model resolving an image.

A diffusion model starts with noise and progressively refines it into a coherent picture, guided by the prompt and the model's learned understanding of structure.

The agent starts with an empty project — noise — and iteratively resolves it into working software. The specifications are the prompt. Architecture patterns are the learned structure. Each iteration of the execution loop is a denoising step.

Denoising stepOne iteration of the execution loop
Early steps: broad compositionScaffolding, data models, API shapes
Late steps: fine detailEdge cases, error handling, polish

Why Raw Inputs Beat Synthesized Ones

A natural instinct: take your PRD and architecture docs, synthesize them into one clean "spec" document, then feed that to the agent. This is a mistake.

The Data Processing Inequality from information theory explains why. For any processing chain X → Y → Z:

I(X; Z) ≤ I(X; Y)

Processing can only preserve or destroy information. It cannot create it. When you synthesize documents X and Y into document Z, you haven't added information. You've compressed it — lossily.

What gets lost

  • Redundancy — The same concept described two different ways in X and Y gives the agent two chances to understand it. Z picks one description and discards the other.
  • Context — The way information is structured in the original document carries meaning. Reorganizing it loses structural context.
  • Edge cases — Synthesis tends to capture the main points and drop the details. The details are where bugs live.
  • Implicit information — What's adjacent to what in the original documents encodes relationships. Synthesis disrupts these.

When synthesis IS valuable

The exception: when the synthesis step adds genuinely new information that wasn't in either source document.

  • Human judgment — Resolving ambiguities between X and Y that the agent can't resolve alone
  • Derived constraints — "Given the data model AND the API contract, the batch size must be < 1000" (neither doc says this alone)
  • Conflict resolution — When X and Y contradict each other, a human choosing which is correct adds information

If your synthesis step is just reformatting or combining, skip it. Feed the raw documents. The agent handles multiple inputs better than you think.
