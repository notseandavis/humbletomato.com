Thought Experiment

The Telephone Game

Every time you pass information through an LLM, you lose something. What does that mean for how we build AI workflows?

The Setup

You have a PRD and an architecture doc. You want an AI agent to generate code. Two options:

Option A:  PRD + Architecture  -->  Code Agent  -->  Code

Option B:  PRD + Architecture  -->  Synthesis Agent  -->  "Unified Spec"  -->  Code Agent  -->  Code

Option B feels more organized. You're "cleaning up" the inputs, making them "easier for the agent." Surely a polished, unified spec produces better code than two messy documents?

Information theory says no. Option B is strictly worse.

The Telephone Game

Remember the game? A message passes through a chain of people. Each person hears it, processes it, and passes it along. The message at the end is always degraded. Never improved.

Every LLM call works the same way. When the synthesis agent reads your PRD and architecture doc, it:

  • Summarizes some sections, losing detail
  • Paraphrases others, shifting meaning ("must" becomes "should")
  • Drops things it considers "irrelevant" that matter for implementation
  • Resolves ambiguities by guessing, sometimes wrong
  • Loses positional context (which section referenced which)

The synthesized spec contains at best the same information as the originals. Usually less. This isn't a guess. It's a mathematical law.

The Data Processing Inequality

Shannon's Data Processing Inequality states: processing cannot create information, only preserve or destroy it. If you have data X and process it into Z:

Information in Z about the original ≤ Information in X about the original

No amount of clever prompting changes this. The synthesis agent cannot know what the code agent will need. It compresses based on general heuristics, not downstream requirements. Details it discards as noise might be exactly what the code agent needs to handle an edge case.

It Gets Worse With Each Step

Each additional LLM step in a chain compounds the loss:

Step 1:  PRD + Arch  -->  "Unified Spec"     (some information lost)
Step 2:  "Unified Spec"  -->  "Task Breakdown"  (more lost)
Step 3:  "Task Breakdown"  -->  Code             (even more lost)

Research on LLM chains calls this "cascading drift." An error or omission in step 1 propagates to step 2, where it gets amplified. By step 3, the code agent is working from a degraded version of your original intent.

The direct path has one step. One chance for loss. Every intermediate step is another round of telephone.

But What About "Cleaning Up" Messy Docs?

This is the intuition that leads people astray. Messy docs feel like they need preprocessing. But modern LLMs handle messy, real-world documents better than you'd expect. They can extract what they need from imperfect inputs.

When you "clean up" docs through an LLM, you're not removing noise. You're removing information you can't distinguish from noise. The code agent might have used that "noise" to make a better decision.

When Intermediate Steps DO Help

The inequality has one exception: processing that adds genuinely new information.

Helps (adds information)Hurts (loses information)
Human resolves conflict between PRD and architectureLLM "unifies" two documents into one
Domain expert adds implicit constraintsLLM summarizes for "clarity"
Generating tests (new artifact, new information)Restructuring docs to "help the agent"
Validation that flags contradictionsParaphrasing requirements

Human synthesis can add value because humans bring knowledge not in the documents. LLM synthesis can only rearrange what's already there, and it always loses something in the process.

The Spec Template as Validation Point

This is why the spec template in the Matrix Methodology works despite the telephone game principle. The template isn't an LLM synthesizing two documents into one. It's a human connecting dots between unstructured context (PRD, architecture) and specific implementation decisions.

When you write a spec template that maps "user enters address" to "REST controller with @Valid request body," you're not compressing information. You're adding the mapping that didn't exist in either source document. You can see how the different concepts resolve together. That's new information.

It's not theoretically necessary. You could feed raw PRD + architecture directly and get working code. But the spec template is a validation checkpoint: if the mapping doesn't make sense when you write it down, you've caught a problem before the agent touches any code. That's worth the extra step.

The key distinction: a human creating a spec template is resolving ambiguity and adding implementation decisions. An LLM "unifying" two documents is compressing and paraphrasing. One adds signal. The other loses it.

The Practical Takeaway

Feed your AI agents the richest, rawest context you can. Don't preprocess "for clarity." Don't synthesize "for convenience." Every transformation step between your intent and the generated output is a round of telephone.

If your documents fit in the context window (and with 200K token windows, they usually do), the optimal pipeline is the shortest one:

Your intent  -->  Agent  -->  Output

Fewer steps. Richer context. Less telephone.
