What the Model Already Knows

You don't need to teach it how to code. You need to tell it what to build.

The Baseline Is Massive

The model already knows how to write C. And Java. And JavaScript, Python, Rust, Go, Ruby, TypeScript, Kotlin, Swift. It knows assembly. It knows binary encoding. It's read more code than any human ever will.

But it's not just languages. It knows:

  • How to build a REST API, a GraphQL schema, a WebSocket server
  • How to structure a React app, a Vue app, an Angular app
  • How to design a relational schema, a document store, a cache layer
  • How to write a CLI tool, a daemon, a browser extension
  • How to handle auth, sessions, encryption, rate limiting
  • How to lay out a dashboard, a form, a checkout flow
  • How UX patterns work — navigation, feedback, error states, accessibility
  • How to write tests, CI pipelines, Dockerfiles, Kubernetes manifests

All of this is free. It's baked into the weights from training. You never need to explain any of it.

What It Doesn't Know

What the model doesn't know is what you want. That's the only missing piece.

Already in the model:
  ✓ How to implement OAuth2 with JWT
  ✓ How to implement OAuth2 with sessions
  ✓ How to implement OAuth2 with PKCE
  ✓ How to implement SAML
  ✓ How to implement API keys
  ✓ ... hundreds more auth patterns

Not in the model:
  ✗ Which one YOUR system uses

The model can build anything. It just doesn't know which thing to build. That's what your spec provides — not the how, but the what and the which.

The Implication

Your spec doesn't need to explain how pagination works, or how to structure a React component, or how to write a SQL migration. The model already knows all of that better than most engineers do.

Your spec needs to say: paginate with cursor-based pagination, 25 items per page, sorted by created date descending. The implementation knowledge is free. The decisions are what cost you.
